# Retry Strategy at Integration Boundaries

Retry behavior at integration boundaries requires more deliberate design than retry behavior within a single system. The external system is not under operational control. Its failure modes, rate limits, and recovery characteristics are documented incompletely and observed empirically. Retry configurations that work correctly against an external system in normal operation can behave destructively during that system's degradation event.

The fundamental distinction is between transient and permanent failures. Transient failures — network timeouts, temporary unavailability, rate limit responses — should trigger retry with appropriate backoff. Permanent failures — authentication errors, schema validation rejections, resource not found — should not. Conflating these produces retry storms against systems that cannot serve the request regardless of how many times it is retried.

Exponential backoff with jitter is the correct default for transient failures at external boundaries. Fixed-interval retry is appropriate only for internal queues where the consumer controls the backoff ceiling. Linear backoff sits between the two and is useful when the failure mode is rate limiting and the retry schedule needs to be more predictable than exponential. The jitter component is not optional in high-concurrency environments — synchronized retry schedules from multiple callers produce correlated load spikes that exacerbate the original degradation.

The retry budget is as important as the retry schedule. Unbounded retries with correct backoff will eventually exhaust a shared resource or fill a queue with stale requests. Per-job and per-integration retry budgets define a ceiling on total retry attempts before the request is dead-lettered. The dead-letter path is not a failure state to be avoided — it is an expected output of the retry system for requests that cannot be delivered within the budget.

External rate limits deserve special handling. A 429 response from an external system should not trigger the same backoff schedule as a 503. Rate limit responses typically include retry-after headers or documented backoff windows. Respecting these explicitly rather than applying a generic exponential backoff demonstrates good integration citizenship and avoids triggering rate limit escalation by the external provider.
